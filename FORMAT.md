# TopoDB on-disk format (v6)

This is the durable contract for a TopoDB redb file. Code wins if it differs
from this document. Nine fixtures under `crates/topodb/tests/fixtures/` pin
this contract:

- `v1.redb`, `v2.redb`, `v2-workload.redb` — FROZEN pre-v3 migration corpora,
  none regenerable by this build and none may be deleted or byte-modified
  (see `format_fixture.rs`'s module doc comment).
- `v3.redb` — the native v3 fixture, last regenerated during the
  storage-format-v4 plan's Task 5 (so it already dual-writes the v4
  `vectors`/`embedding_ref` tables). `FORMAT_VERSION` is 4 everywhere on this
  branch, so no build here can produce a fresh v3 file anymore; `v3.redb` is
  now FROZEN too, exercised by `v3_fixture_opens_and_reads`.
- `v3-legacy.redb` — a FROZEN, genuinely pre-v4 v3 file recovered byte-for-byte
  from commit `a3711eb`. It predates the `vectors`/`embedding_ref`/
  `vector_dims` tables entirely, so it is the one corpus that proves
  migration creates and populates them from scratch rather than merely
  overwriting already-present rows. Never regenerable.
- `v3-workload.redb` — a FROZEN, larger (200-memory) v3 corpus captured before
  postings maintenance was rewired to the v4 chunked layout, so it is the one
  corpus that exercises the postings-rechunking pass at multi-chunk scale.
  Generated by `generate_v3_workload.rs`'s one-shot `#[ignore]`d test, never
  re-run now that no build on this branch can write single-row-per-term v3
  postings again.
- `v4.redb` — the native v4 fixture, FROZEN since the v5 flip (no build on
  this branch can stamp a fresh file at 4), exercised by
  `v4_fixture_opens_and_reads` as the v4 migration-input file.
- `v5.redb` — the native v5 fixture, FROZEN since the v6 flip (no build on
  this branch can stamp a fresh file at 5), exercised by
  `v5_fixture_migrates_to_v6_and_reads` as the v5 migration-input file.
- `v6.redb` — the native v6 fixture, and the ONLY fixture this build can
  actually regenerate (`format_fixture.rs::regenerate_v6_fixture`).

## Version and migration

`storage::FORMAT_VERSION` is **6**. v6 adds exactly **one new table**,
`LABEL_INDEX` (`(label_id, scope_id, node_id) -> slot`, see "Tables," below) —
a derived index over each node's already-interned label and scope, maintained
incrementally by `apply_op` (insert on `CreateNode`, remove on `RemoveNode`;
labels are immutable, so there is no update case) and rebuilt wholesale by
`rebuild_state_from_ops`. No existing table's layout changes. A pre-v6 file's
`LABEL_INDEX` is built by `migrate_v6::migrate_v5_to_v6`: a single scan of
`NODES`, decoding each row's already-interned `label`/`scope` fields straight
off `disk::NodeRecordDiskV3` (no `Dicts`/`ScopeRegistry` resolution needed —
those ids are already exactly what the key wants) and inserting one
`LABEL_INDEX` row per node. This is deliberately the simplest hop in the
file's history: one derived table, one scan, no re-encoding of anything else,
no dual-write era.

v5 changed **no table layout** relative to
v4 — only the `prop_index` key scheme: `Str` values are keyed by their
*normalized* form (`prop_index::normalize_str` — whitespace-trimmed/collapsed,
Unicode-lowercased) instead of their raw bytes, making equality-index probes
case- and whitespace-insensitive (`Db::nodes_by_prop` restores byte-exact
semantics with a record-level post-filter; `Db::nodes_by_prop_normalized`
exposes the relaxed match). v5 also versioned the FTS analyzer: postings are
built from the v1 pipeline (split on non-alphanumeric → camelCase split →
Unicode lowercase → Snowball English stem, `fts::tokenize`), stamped in META
as `"fts_analyzer_version"` — a file whose stamp is absent or different gets
its FTS tables drained and rebuilt on open, so postings always agree with
what a query tokenizes to. Both rebuilds are driven by `ensure_index_spec`'s
META checks (see below), not by the version-match arms; the v4→v5 arm only
stamps the version so pre-v5 builds refuse the file instead of silently
missing every Str probe.

`Storage::open_with_options` opens (or creates) every table, reads META
`"format_version"`, and dispatches on it — all inside ONE redb write
transaction, committed once at the very end:

| stored version | on open |
|---|---|
| absent (brand-new file) | stamp `FORMAT_VERSION` (6) directly — `storage.rs: Storage::open_with_options`, `None` arm |
| 6 | open directly, no rewrite — `Some(6) => {}` |
| 5 | `migrate_v6::migrate_v5_to_v6` (one NODES scan, builds `LABEL_INDEX`), stamp 6 — `Some(5)` arm |
| 4 | `migrate_v6::migrate_v5_to_v6`, stamp 6 directly (this arm jumps straight from 4 to the current `FORMAT_VERSION`, so it must do v6's table work itself rather than falling through to the `Some(5)` arm, which only runs for a file genuinely stamped 5; the PROP_INDEX rebuild still happens separately, in `ensure_index_spec`, via the norm-version check) — `Some(4)` arm |
| 3 | `migrate_v4::migrate_v3_to_v4` (postings pass runs: `postings_already_chunked = false`), then `delete_table(EMBEDDINGS)`, `migrate_v6::migrate_v5_to_v6`, stamp 6 — `Some(3)` arm |
| 2 | `migrate_v3::migrate_v2_to_v3` (now also dual-writes `vectors`/`embedding_ref` as it walks each migrated node), immediately followed in the SAME open by `migrate_v4::migrate_v3_to_v4` (postings pass skipped: `postings_already_chunked = true`), then `delete_table(EMBEDDINGS)`, `migrate_v6::migrate_v5_to_v6`, stamp 6 — `Some(2)` arm, chained in one call |
| 1 | `migrate::migrate_v1_to_v2` (stamps 2), then `migrate_v3::migrate_v2_to_v3`, then `migrate_v4::migrate_v3_to_v4` (`postings_already_chunked = true`), then `delete_table(EMBEDDINGS)`, `migrate_v6::migrate_v5_to_v6`, stamp 6 — `Some(1)` arm, chained in one call |
| > 6 | `TopoError::UnsupportedFormat { found, supported: 6 }` — `Some(found) if found > FORMAT_VERSION` |
| anything else (currently only 0, which can't occur since real versions start at 1) | `TopoError::Encoding` — `Some(found)` catch-all arm |

Every arm above except the fast `Some(6) => {}` path calls
`migrate_v6::migrate_v5_to_v6` — not just the `Some(5)` arm — because arms for
versions further back stamp `FORMAT_VERSION` directly rather than falling
through a chain of one-version-at-a-time steps; each must therefore perform
v6's table work itself. `migrate_v5_to_v6` is idempotent (re-inserting the
same key/value is a no-op), so calling it unconditionally on every non-fast
path is safe regardless of how many other migrations ran first in the same
open.

Because table creation, migration, and the version stamp all happen inside the
single write transaction `open_with_options` opens at its top and commits once
at the bottom (`storage.rs`, around `let tx = s.db.begin_write()...` through
`tx.commit()`), a crash or kill at any point before that one commit leaves the
file at exactly its prior on-disk version — including mid-way through the
chained v1→v2→v3→v4 path. There is no partially-migrated state observable on
reopen.

`EMBEDDINGS` (`storage.rs`) is v3-and-earlier only: it is NOT part of the
tables `open_with_options` opens unconditionally (see "Tables," below). Every
pre-v4 match arm opens it itself, `migrate_v4::migrate_v3_to_v4`'s vectors
pass reads it (never writes it), and the caller — `open_with_options`, not
`migrate_v3_to_v4` — deletes it via `WriteTransaction::delete_table` once that
pass has drained everything worth keeping out of it. A v4-native file never
has this table at all.

Migration rewrites are scoped precisely:
- **v1 → v2** (`migrate::migrate_v1_to_v2`, frozen): decodes the old
  embedded-embedding `NodeRecordV1`/`EdgeRecordV1` rows, externalizes
  embeddings into `EMBEDDINGS` (still ULID-keyed at this point), interns
  label/edge-type/prop-key strings into `DICT`, and rewrites `NODES`/`EDGES`
  via `disk::node_to_disk`/`disk::edge_to_disk` (the frozen v2 ULID-keyed
  encode shape). Never touches `OPS`, `POSTINGS`/`FTS_DOCS`/`FTS_STATS`,
  `COUNTERS`, or unrelated `META` keys.
- **v2 → v3** (`migrate_v3::migrate_v2_to_v3`): consumes v2 `NODES`/`EDGES`
  (decoded through the frozen `NodeRecordDiskV2`/`EdgeRecordDiskV2` twins) plus
  a pre-drain snapshot of the old ULID-keyed `EMBEDDINGS`/`COUNTERS` rows, and
  produces: dense node/edge slots (`NODE_SLOTS`/`NODE_IDS`/`EDGE_SLOTS`/
  `EDGE_IDS`), slot-rekeyed `NODES`/`EDGES`/`EMBEDDINGS`/`COUNTERS`, a seeded
  `SCOPES` registry, `OUT_ADJ`/`IN_ADJ` adjacency built from every edge row,
  `PROP_INDEX` rebuilt from the caller's `IndexSpec`, and `POSTINGS`/
  `FTS_DOCS`/`FTS_STATS` fully **rebuilt** (not transcoded) via the same
  `fts_update` the write path uses — the pre-W2b v2 FTS layout is
  byte-incompatible with v3's scope-id/slot keying, so it is drained and
  regenerated once per node instead. `DICT` rows carry forward unchanged
  (label/type/prop-key ids are stable across v2→v3). `OPS` is untouched;
  `META` is untouched except `"next_node_slot"`/`"next_edge_slot"`, which this
  step clears and then repopulates as it allocates fresh slots (a v2 file has
  no slot counters yet — slots don't exist before v3). **v4 addition**: for
  each migrated node that carried an old embedding, this step now ALSO
  dual-writes it into `vectors`/`embedding_ref` — interning the model name
  into `DICT` (`DictKind::Model`) and pinning/checking its dim via
  `check_or_pin_dim`, mirroring `apply_op`'s `SetEmbedding` arm exactly
  (`migrate_v3.rs: migrate_v2_to_v3`, the `old_embeddings.get(...)` block).
  Without this, a v1/v2-origin file would carry its embeddings only in the
  still-present `EMBEDDINGS` table and never reach the v4 `vectors`/
  `embedding_ref` tables `Db::search_vector` reads exclusively — this was
  caught by `format_fixture.rs` during the v4 plan (Task 5) and fixed here.
- **v3 → v4** (`migrate_v4::migrate_v3_to_v4`, new this version): runs two
  independent passes inside the caller's already-open write transaction.
  - *Vectors pass* (always runs): walks `EMBEDDINGS` (slot-keyed at this
    point), joins each row against `NODES` for its scope, interns the model
    name into `DICT` (`DictKind::Model`), pins/checks the model's dim via
    `check_or_pin_dim`, and writes into `vectors`/`embedding_ref` via
    `vector_store::put_vector` — the same helper the live `SetEmbedding` write
    path uses. Does not delete `EMBEDDINGS` itself (the caller does, once its
    own borrow of the table has ended) and does not touch `FTS_DOCS`/
    `FTS_STATS` (unchanged across v3→v4 — only `POSTINGS`'s key/value shape
    changes).
  - *Postings pass* (conditional on `postings_already_chunked`): re-chunks
    every single-row-per-term v3 `POSTINGS` row into the v4 chunked block
    layout, replaying each row's `(slot, tf)` entries ascending-by-slot
    through the SAME incremental `fts::set_posting` the live write path uses,
    so the result is byte-identical to what incremental maintenance would
    have produced. The `Some(3)` arm passes `false` (a genuine v3 file only
    ever wrote single-row postings); the `Some(2)`/`Some(1)` chains pass
    `true`, since `migrate_v2_to_v3`'s FTS rebuild — running earlier in the
    SAME open, under THIS build's current `fts_update` — already wrote
    chunked rows directly, so re-chunking would be not just redundant but
    would misparse already-chunked bytes as the old unframed shape. **RAM
    note:** when it runs, this pass first groups every old v3 `POSTINGS`
    row's decoded entries by `(scope_id, term)` in an in-memory `HashMap`
    before writing anything back, so it briefly holds the entire old
    `POSTINGS` table's decoded content in RAM. In practice this is bounded
    by the same cost v3's own incremental indexing already paid to build
    that table in the first place; a file with an unusually large or dense
    vocabulary may see a brief RAM spike during this one-time migration.

  **Key-shape discrimination is a process invariant, not a code-enforced
  one.** A v3-native `POSTINGS` row (`scope_id ++ term`, no chunk suffix) and
  a v4 chunked row (`scope_id ++ term ++ chunk`) are not always
  byte-distinguishable by key alone, so the caller states which shape a file
  holds explicitly via `postings_already_chunked` rather than sniffing
  content. The premise behind `Some(3) => false` — "no RELEASED build ever
  writes chunked postings under `format_version == 3`" — held only for
  released builds; this branch's own history briefly violated it
  (commits 9b3d5a7 through 70bcd09 stamped 3 while already writing chunked
  postings, mid-branch, before the format flip landed). Should a
  chunked-under-3 file ever reach the `Some(3)` arm anyway, a
  byte-layout-coincidental backstop catches it: `decode_v3_posting_value`'s
  trailing-bytes check rejects a chunked block's leading
  `POSTINGS_BLOCK_FORMAT_V0` (`0x00`) byte, which decodes as varint
  `count == 0` (so the entry loop reads nothing) followed by non-empty
  trailing bytes (a stored chunk is never empty) — the migration fails LOUDLY
  with `TopoError::Encoding`, aborting the whole open with the file left
  byte-intact, rather than silently corrupting the table. This backstop is
  pinned by `chunked_postings_under_version_3_fail_migration_loudly_not_silently`
  (`storage.rs` tests); a future postings-codec change could silently defeat
  it, which is exactly what that test exists to catch. See
  `migrate_v4.rs`'s module doc comment ("Postings key-shape discrimination")
  for the full rationale.

  **Idempotency.** The `Some(2)`/`Some(1)` chains reach `migrate_v3_to_v4`
  AFTER `migrate_v2_to_v3` has already dual-written `vectors`/`embedding_ref`
  from the very same (by-then slot-keyed) `EMBEDDINGS` rows the vectors pass
  also scans. Rather than thread an "already written" flag through the call
  chain, the vectors pass runs unconditionally and overwrites: `check_or_pin_dim`
  seeing the same `(model, dim)` twice is `Ok(())` both times, and
  `put_vector` writing the identical `(model, scope, slot, vector)` a second
  time produces a byte-identical row — a harmless redundant write.

  **Per-model dim policy failure (v4 semantics change).** The v3-and-earlier
  RAM slab pinned an embedding model's dimension per-`(model, scope)`, so a
  v3 file could legally record one model at two different dimensions across
  two different scopes. v4 pins dimension **per model only**
  (`check_or_pin_dim`, permanent, cross-scope, cross-batch — see "Records,"
  below). A file whose historical `EMBEDDINGS` rows hit this case fails the
  WHOLE migration with `TopoError::Rejected` (naming the model and both
  dims), never `Encoding` — this is legal upstream data meeting a new v4
  policy, not file corruption. Pinned by
  `v3_file_with_one_model_two_dims_across_scopes_rejects_migration_not_encoding`
  (`storage.rs`, exercising the real `Some(3)` open arm end to end) and
  `one_model_two_dims_across_scopes_rejects_not_encoding` (`migrate_v4.rs`,
  exercising the migration function directly).

- **v5 → v6** (`migrate_v6::migrate_v5_to_v6`, new this version): a single
  scan of `NODES`, no other table read or written (besides `LABEL_INDEX`
  itself). Each row's `disk::NodeRecordDiskV3.label`/`.scope` fields are
  already the interned ids `LABEL_INDEX`'s key wants, so this step needs no
  `Dicts`/`ScopeRegistry` lookups — just a raw decode-and-insert per node.
  Idempotent, so every historic version arm above calls it unconditionally
  (see the version-dispatch table's note). By far the smallest migration hop
  in this document's history: no dual-write era, no re-encoding of an
  existing table, no key-shape discrimination.

Redb free space means the on-disk file size need not shrink even where
logical row bytes do — e.g. a v1 file's per-node embedded embeddings become
smaller externalized rows after migration, but the file itself may stay the
same size or grow, since redb reuses/retains pages via its own free list
rather than truncating. Explicit compaction (`redb::Database::compact`, used
by `regenerate_v4_fixture` when shipping a fixture) is a separate, deliberate
step never run automatically by `open_with`.

## Tables

`storage.rs: Storage::open_with_options` unconditionally opens **twenty-one**
tables on every open (verified from its `existing` block, `OPS` through
`META`) — this is the complete set. `EMBEDDINGS` is deliberately NOT in this
list (see "Version and migration," above) — it exists only mid-migration on a
pre-v4 file and is gone by the time the opening write transaction commits.

| table | key | value | defined in |
|---|---|---|---|
| `ops` | u64 sequence (native redb `u64` key type — LE bytes, but ordered numerically by redb's own `Key::compare`, not lexicographically) | raw postcard `Op` (never interned or framed) | `storage.rs: OPS` |
| `meta` | UTF-8 string | key-specific bytes (see below) | `storage.rs: META` |
| `nodes` | 8-byte BE node slot | framed postcard `disk::NodeRecordDiskV3` | `storage.rs: NODES` |
| `edges` | 8-byte BE edge slot | framed postcard `disk::EdgeRecordDiskV3` | `storage.rs: EDGES` |
| `counters` | 8-byte BE node slot | plain postcard `counters::AccessStats` (not framed) | `storage.rs: COUNTERS` |
| `postings` | `[scope_id:4 BE][term UTF-8][chunk:4 BE]` (chunked, v4) | framed `[block_format:u8][count varint]` then `count` `(slot_delta, tf)` varint pairs | `storage.rs: POSTINGS`, `fts.rs` |
| `fts_docs` | 8-byte BE node slot | plain postcard `u32` token length (not framed) | `storage.rs: FTS_DOCS`, `fts.rs` |
| `fts_stats` | 4-byte BE scope id | plain postcard `(u64, u64)` = `(doc_count, total_len)` (not framed) | `storage.rs: FTS_STATS`, `fts.rs` |
| `vector_dims` | 4-byte BE `DictKind::Model` dict id | 4-byte **LE** `u32` dim (not framed) | `storage.rs: VECTOR_DIMS` |
| `vectors` | `[model:4 BE][scope:4 BE][slot:8 BE]` (16 bytes) | framed postcard `Vec<f32>` | `vector_store.rs: VECTORS` |
| `embedding_ref` | 8-byte BE node slot | postcard `(u32, u32)` = `(model, scope)` (not framed) | `vector_store.rs: EMBEDDING_REF` |
| `dict` | `[kind:u8][id:u32 BE]` (5 bytes) | UTF-8 interned string (not framed) | `dict.rs: DICT` |
| `scopes` | 4-byte BE scope id | 17-byte `scope_key` | `scopes.rs: SCOPES` |
| `node_slots` | 16-byte BE node ULID | 8-byte LE slot | `slots.rs: NODE_SLOTS` |
| `node_ids` | 8-byte BE node slot | 16-byte BE node ULID | `slots.rs: NODE_IDS` |
| `edge_slots` | 16-byte BE edge ULID | 8-byte LE slot | `slots.rs: EDGE_SLOTS` |
| `edge_ids` | 8-byte BE edge slot | 16-byte BE edge ULID | `slots.rs: EDGE_IDS` |
| `out_adj` | `[slot:8][edge_type:4][chunk:4]` (16 bytes) | framed adjacency block | `adj.rs: OUT_ADJ` |
| `in_adj` | `[slot:8][edge_type:4][chunk:4]` (16 bytes) | framed adjacency block | `adj.rs: IN_ADJ` |
| `prop_index` | `[prop_key:4][tag:1][canonical value][slot:8]` | empty (existence-only key) | `prop_index.rs: PROP_INDEX` |
| `label_index` | `[label_id:4 BE][scope_id:4 BE][node_id:16 BE]` (24 bytes, `storage.rs: label_index_key`) | 8-byte native `u64` node slot (not framed) | `storage.rs: LABEL_INDEX` |

`node_key`/`edge_key` (16-byte BE ULID keys, `storage.rs`) are NOT part of the
live v4 read/write path — they remain in use only by `migrate.rs`'s frozen
v1→v2 step and by `migrate_v3.rs`, which reads OLD v2-keyed
`EMBEDDINGS`/`COUNTERS` rows by ULID before re-keying them to slots.

`embeddings` (`storage.rs: EMBEDDINGS`) is documented here, in the migration
history, only: it is the v3-and-earlier cold-vector table — 8-byte BE node
slot key, framed postcard `(String, Vec<f32>)` value (model name, vector) —
kept in the source solely as the table definition `migrate_v4.rs`'s vectors
pass reads and the pre-v4 `open_with_options` arms still write to mid-chain,
before the migrating open deletes it. No v4-native file has this table.

## Key and value encodings

**Dense slots** (`slots.rs`): `NODE_SLOTS`/`EDGE_SLOTS` map a 16-byte BE ULID
key (`id_key`) to an 8-byte **LE** slot value; `NODE_IDS`/`EDGE_IDS` are the
reverse mapping, 8-byte BE slot key (`slot_key`) to 16-byte BE ULID value.
Allocation (`slots.rs: alloc`) is monotonic and never reuses a freed slot:
`META` counters `"next_node_slot"`/`"next_edge_slot"` (8-byte LE u64) hold the
next slot to hand out. `NODES`/`EDGES`/`COUNTERS`/`FTS_DOCS` are all keyed by
the 8-byte BE slot (`storage.rs: slot_key`); `EMBEDDING_REF` is too (see
"Vectors," below).

**Dictionary** (`dict.rs`): key is `[kind:u8][id:u32 BE]` (5 bytes,
`dict::key`); value is the raw UTF-8 interned string (not framed). The kind
byte is an **APPEND-ONLY** registry (`dict::DictKind`): `0x00` Label, `0x01`
EdgeType, `0x02` PropKey, `0x03` Model (v4, `dict.rs: DictKind::Model` —
embedding model names, interned so `vector_dims`/`vectors` can key by a
stable `u32` id rather than the raw string; added at the end of the
discriminant list, never reordered or reused). Dictionary rows are derived,
append-only state — `intern` never removes or renumbers an id. An id absent
from `Dicts::resolve` while decoding is corruption and returns
`TopoError::Encoding`.

**Scopes** (`scopes.rs`): key is a 4-byte BE `u32` registry id; value is the
fixed-width 17-byte `scope_key` (`storage.rs: scope_key`) — 1-byte tag
(`0x00` Shared, all-zero remainder; `0x01` Id, followed by the 16-byte BE
scope ULID). Row `0` is always `Scope::Shared`, seeded by `scopes::seed_shared`
on every `open_with` and by every rebuild/migration path;
`ScopeRegistry::load_table_for_rebuild` refuses to load a table where id `0`
isn't `Shared`. Interning is append-only, same doctrine as the dictionary.

**Label index** (v6, `storage.rs: LABEL_INDEX`/`label_index_key`): key is
`[label_id:4 BE][scope_id:4 BE][node_id:16 BE]` (24 bytes); value is the raw
`u64` node slot. Purely derived state — reconstructible from `NODES` alone
(see "Version and migration"'s v5→v6 description, and "Rebuild determinism,"
below) — maintained incrementally by `apply_op`: inserted once a `CreateNode`
has allocated its slot, removed on `RemoveNode` using the label/scope read
straight off the just-removed row (labels are immutable — there is no
`SetNodeProps`-driven update case). The `node_id` tail means every row
sharing a `(label_id, scope_id)` prefix sorts in mint-time order (a ULID's
leading bits are a timestamp), so a bounded prefix scan over that key range
visits a label's nodes within a scope oldest-first — the property the
label-scoped read path (`Storage::load_nodes_by_label`/
`load_nodes_by_label_newest`, wired up in Task 8) leans on.

**Value frame** (`codec.rs`): every `nodes`, `edges`, `vectors`, `postings`
chunk, and adjacency (`out_adj`/`in_adj`) block value is wrapped in a 1-byte
codec-tag frame (`codec::frame_value`/`unframe_value`) — verified by grepping
every `frame_value`/`unframe_value` call site in `src/`. `counters`,
`fts_docs`, `fts_stats`, `dict`, `scopes`, `prop_index`, `vector_dims`,
`label_index`, and `embedding_ref` are plain (unframed) postcard or raw
bytes; `ops` and `meta` are untouched raw postcard/bytes. The codec byte is
an **APPEND-ONLY**
registry: `0x00` (`CODEC_RAW`) uncompressed postcard; `0x01` unused (skipped
in the numbering, available for a future codec); `0x02` (`CODEC_LZ4`) an
`lz4_flex` size-prepended block. `frame_value` only attempts lz4 for raw
payloads `>= 512` bytes, and keeps the compressed form only if it's `<= 90%`
of the raw size — otherwise it falls back to raw. `unframe_value` rejects an
lz4 frame whose declared decompressed size exceeds `256 MiB`. `lz4_flex` is
pure Rust; no C zstd dependency is used. `EMBEDDINGS` (v3-and-earlier only)
also used this frame for its `(String, Vec<f32>)` value — see "Tables,"
above.

**Adjacency** (`adj.rs`): key is `[slot:8 BE][edge_type:4 BE][chunk:4 BE]`
(16 bytes, `adj::out_adj_key`) — `(slot, edge_type)` addresses a bounded range
of chunks, `chunk` is a dense split counter (`adj::CHUNK_SPLIT_TARGET` = 8 KiB
encoded-size target per chunk; `adj_insert` splits a chunk in half once it
would exceed that). The framed block payload (`adj::encode_block`/
`decode_block`) is:
1. `block_format: u8` — **APPEND-ONLY** registry, currently only `0x00`
   (`ADJ_BLOCK_FORMAT_V0`); any other byte is rejected as
   `"unknown adjacency block format"`.
2. `count: varint`
3. `count` entries, each: `target_delta: varint` (delta from the previous
   entry's `target`, first entry relative to 0 — entries are required
   target-sorted, `encode_block` rejects an out-of-order slice), `edge: varint`
   (edge slot), `scope: varint` (scope registry id, cast from `u32`),
   `valid_from: varint` (zigzag-encoded `i64`), then a 1-byte `valid_to` tag
   (`0` = `None`; `1` = `Some`, followed by a zigzag `varint`).

**Prop index** (`prop_index.rs`): key is `[prop_key:4 BE][tag:1][canonical
value bytes][slot:8 BE]` (`prop_index::index_key`); value is empty
(existence-only row). The value tag registry (`prop_index::value_bytes`,
fixed — not one of the append-only byte registries called out in the
Evolution policy below, but stable in practice since it mirrors
`index::IndexValue`'s variants): `0` Str (v5: UTF-8 bytes of the NORMALIZED
value — `prop_index::normalize_str`: whitespace-trimmed, internal runs
collapsed to one space, Unicode-lowercased; the raw value lives only in the
node record. Versioned by the `"prop_index_norm_version"` META stamp — a file
whose stamp differs from `prop_index::PROP_INDEX_NORM_VERSION` gets its whole
PROP_INDEX drained + rebuilt by `ensure_index_spec` on open), `1` Int (sign-flip
trick: `(v as u64) ^ (1 << 63)`, BE 8 bytes — preserves signed ordering in an
unsigned lexicographic key space), `2` Bool (1 byte, `0`/`1`), `3` Bytes (raw),
`4` DateTime (same sign-flip as Int, BE 8 bytes). `PropValue::Float` is never
indexed (`IndexValue::of` returns `None` for it). `lookup` range-scans the
`[prop_key][tag][value]` prefix and then filters by exact key length
(`prefix.len() + 8`) — load-bearing, since a longer Str/Bytes value sharing a
shorter one's byte prefix (`"ab"` vs `"abc"`) would otherwise fall inside the
same range scan.

**Vectors** (v4, `vector_store.rs`): clustered embedding storage, replacing
the v3-and-earlier slot-keyed `EMBEDDINGS` cold table plus its in-RAM
per-`(model, scope)` slab index (both deleted this version — see "The
deleted v3 vector index," below).

- `vectors` key (`vector_store::vector_key`): a fixed-width 16-byte
  `[model:4 BE][scope:4 BE][slot:8 BE]` — the `(model, scope)` prefix sorts
  first, so every row in one model+scope cluster is one contiguous,
  boundedly-scannable range (`vector_store::vector_prefix` bounds that scan).
  Fixed width means no trailing-length check is needed to exclude a longer
  sibling key, unlike `prop_index`'s variable-length keys. Value: framed
  postcard `Vec<f32>`.
- `embedding_ref` key: 8-byte BE node slot (`vector_store::slot_key`). Value:
  postcard `(u32, u32)` = `(model, scope)`, NOT framed — a fixed-size pair
  never crosses the frame's 512-byte lz4-attempt threshold, so wrapping it
  would be pure overhead. Points at a node's CURRENT `(model, scope)`, so a
  re-embed or `RemoveNode` can find (and delete) the OLD `vectors` row in
  O(1) rather than a scan.
- `vector_store::put_vector` writes both rows together: it first reads the
  node's PRIOR `embedding_ref` (if any); a re-embed under a DIFFERENT model
  deletes the old `vectors` row before writing the new one; a re-embed under
  the SAME model lands on the identical key (scope and slot are immutable for
  a node's lifetime) and simply overwrites in place. `remove_vector` clears
  both rows for a slot — a no-op if the node was never embedded (empty-key
  doctrine).
- `vector_store::read_vector_by_slot`'s own corruption contract mirrors
  `storage.rs::read_node`'s: an absent `embedding_ref` row is an ordinary
  `Ok(None)` ("never embedded"); an `embedding_ref` row whose `vectors` row is
  missing is `TopoError::Encoding`, never silently `None` — the two rows are
  always written/removed together by `put_vector`/`remove_vector`.
- Read path: `Db::search_vector` (`vector.rs`) resolves the query model to a
  dict id (an unknown model yields `Ok(vec![])`, no error), then
  `vector_store::search_scan` either range-scans each requested scope's own
  `(model, scope)` cluster, or — when `VectorQuery::candidates` is set —
  looks up each candidate's current embedding via `read_vector_by_slot`
  (O(1) per candidate instead of a range scan). Every scored row runs through
  the same cosine formula the old RAM slab used (`vector_store::cosine`,
  `None` on a zero-norm side — skipped, not rejected); a row whose stored
  vector length disagrees with the query's is likewise skipped. Top-`k`
  candidates are bounded by a min-heap (`vector_store::push_topk`) that
  conservatively retains every element tied at the current boundary score
  until enough strictly-better elements arrive to make the whole tied group
  provably irrelevant — the final `(score desc, NodeId asc)` sort and
  truncation to `k` happens only in `Db::search_vector`, after every
  surviving slot has been resolved to its `NodeId` (slot/creation order is
  NOT `NodeId`/ULID order, so an early tie-break by slot could silently keep
  the wrong side of a tie).

**Vector dims** (v4, `storage.rs: VECTOR_DIMS`): key is a 4-byte BE
`DictKind::Model` dict id; value is a 4-byte **LE** `u32` dimension, written
raw (not framed — a 4-byte value never crosses the frame's compression
threshold). `storage.rs::check_or_pin_dim` pins a model's dimension on its
first appearance (`SetEmbedding`, or a migration folding a historical
embedding into `vectors`): absent → insert; present and equal → `Ok`; present
and different → `TopoError::Rejected` (naming the model id and both dims),
aborting the whole enclosing write transaction. **This is a v4 semantics
change**, not a mechanical rename: the v3-and-earlier RAM slab pinned
dimension per-`(model, scope)`, so the same model could legally carry
different dimensions in different scopes; v4 pins **per model only**,
permanently, across every scope and every batch — see "Version and
migration," above, for the migration-time failure this produces on legacy
data that violates the new rule, and `check_or_pin_dim`'s doc comment
(`storage.rs`) for the write-path enforcement. A zero-dimension embedding is
rejected before it can ever reach `check_or_pin_dim`
(`apply_op`'s `SetEmbedding` arm) — an empty vector would otherwise
permanently pin a model's dimension at 0, after which every real embedding
under that model would reject as a dim conflict forever.

**The deleted v3 vector index.** Through the pre-v4 history of this branch,
`vector.rs` also owned a per-`(model, scope)` in-RAM slab index
(`VectorIndex`/`Slab`) that the applier maintained on every write and
`search_vector` read from, plus dim pre-validation run before each batch
committed. That RAM index, its locking machinery, and its poisoned-lock
policy are gone in v4 — `search_vector` reads the on-disk `vectors`/
`embedding_ref` tables directly (see above), and dim validation now lives
entirely in `check_or_pin_dim`. This has no on-disk footprint of its own
(the RAM slab was never persisted) but is documented here because it is the
reason `EMBEDDINGS` and its old read path are gone: nothing in v4 rebuilds
or maintains an equivalent RAM structure at all.

**Postings** (`fts.rs`, chunked layout — v4): a term's postings under one
scope are split across one or more chunk keys rather than one unbounded row,
so a single hot term's postings never grow into one value that must be fully
rewritten on every touch — the v3 single-row layout made incremental
maintenance quadratic at scale.

- Key (`fts::chunked_posting_key`): `[scope_id:4 BE][term UTF-8][chunk:4 BE]`
  — both the scope prefix and the chunk suffix are fixed-width, so the only
  variable-length part (the term) is always sandwiched between two
  fixed-width fields; two keys can only collide if their terms are
  byte-equal too, since a shorter/longer term produces a different total key
  length.
- Value (`fts::encode_posting_block`/`decode_posting_block`), framed via
  `codec::frame_value` like other framed tables:
  1. `block_format: u8` — an **APPEND-ONLY** registry, currently only `0x00`
     (`fts::POSTINGS_BLOCK_FORMAT_V0`); any other byte is rejected as
     `"unknown postings block format"`.
  2. `count: varint`
  3. `count` entries, each ascending by node slot: `slot_delta: varint`
     (first delta relative to 0) then `tf: varint` (term frequency).
  An empty chunk is never written — the encoder rejects an empty entry slice
  outright, and a chunk that would become empty (last entry removed) has its
  key removed instead (same empty-key doctrine as the prop index and FTS
  stats).
- Maintenance (`fts::set_posting`, `pub(crate)` so `migrate_v4.rs`'s
  re-chunking pass can reuse it): one bounded prefix range scan
  (`fts::term_chunk_keys`) locates the term's chunk keys, then only the
  **last** chunk is decoded first. A new document (the common case — new
  documents carry the highest slots) hits the fast path: append to the
  already-decoded last chunk, decoding exactly ONE chunk regardless of how
  many the term has — O(1) per new document, versus the v3 layout's O(df).
  A slot within the last chunk's own range mutates it in place; a slot
  below it (an update, a removal, or an out-of-order insert) finds its
  covering chunk by binary-searching the earlier chunks' first slots,
  peeked from each block's header (`fts::peek_first_slot` — the first
  entry's delta is relative to 0, so it is the absolute first slot) without
  decoding entries; a slot in the gap between two chunks' ranges lands in
  the earlier chunk. (A probed chunk's framed value is still loaded — and,
  if lz4-framed, decompressed — but its entries are never walked; the
  located covering chunk is then fully decoded and rewritten.) EVERY
  rewrite — append or covering mutation — splits
  at the midpoint entry when the re-encoded chunk exceeds
  `fts::POSTINGS_CHUNK_TARGET` (4 KiB); a mid-list split shifts the later
  chunk keys up one number each (highest first, raw framed bytes moved
  untouched). Chunk numbers are order-bearing, not dense: removing an
  emptied chunk leaves a numbering gap and never renumbers, and readers
  never assume density. None of this changes the stored format — the split
  policy is maintenance behavior; on-disk keys and blocks are exactly the
  shapes above, and v4 files written before this policy need no migration.
  A file written by an earlier v4 build under an edit-heavy history can
  already hold a covering chunk beyond the target; each subsequent rewrite
  of such a chunk splits it once at the midpoint, so oversized legacy
  chunks converge back under the target incrementally rather than
  immediately.
  `fts::read_posting` (used by scoring, which needs every entry) decodes and
  concatenates every chunk in ascending order; `fts::posting_df` (the
  document-frequency fast path `search_text` uses before deciding whether a
  term has any hits at all) sums each chunk's `count` header without decoding
  any entries.
- `fts_docs` (node slot → plain postcard `u32` token count) and `fts_stats`
  (4-byte BE scope id → plain postcard `(doc_count: u64, total_len: u64)`)
  are unchanged from v3 — unframed, maintained by the same `fts::fts_update`
  used by `apply_batch`, `rebuild_state_from_ops`, `ensure_index_spec`'s
  reindex, and `migrate_v3::migrate_v2_to_v3`. A scope whose last document is
  removed drops its `fts_stats` row entirely (same empty-key doctrine).

**META keys** (`storage.rs`, `slots.rs`): `"format_version"` (4-byte LE
`u32`); `"oldest_seq"` (8-byte LE `u64`, absent ⇒ 1, written only by
`compact_ops_through`); `"index_spec"` (postcard `IndexSpec`, both lists
sorted by `(label, prop)` before encoding so declaration order never looks
like a spec change); `"prop_index_norm_version"` (4-byte LE `u32`, v5: the
Str-key normalization scheme PROP_INDEX was last built under — absent or
mismatched ⇒ `ensure_index_spec` drains + rebuilds the index and re-stamps);
`"fts_analyzer_version"` (4-byte LE `u32`, v5: the analyzer pipeline the FTS
tables were last built under — absent or mismatched ⇒ same drain + rebuild +
re-stamp treatment, since postings from a different tokenizer can disagree
with what this build tokenizes a query into);
`"next_node_slot"`/`"next_edge_slot"` (8-byte LE `u64`
monotonic slot counters, `slots.rs`). Legacy Plan-2 keys `"fts_spec"`,
`"fts_doc_count"`, `"fts_total_len"` are read (and, if present, removed) only
by `ensure_index_spec` when opening a Plan-2 file (a file with the legacy
`"fts_spec"` key present — a Plan-1 file has neither `"fts_spec"` nor
`"index_spec"` and never triggers the removal) — no live write path writes
them.

## Records

`disk::NodeRecordDiskV3 { id: NodeId, scope: u32, label: u32, props:
BTreeMap<u32, PropValue> }` and `disk::EdgeRecordDiskV3 { id: EdgeId, scope:
u32, ty: u32, from: u64, to: u64, props: BTreeMap<u32, PropValue>, valid_from:
i64, valid_to: Option<i64> }` are the live record-table shapes (`disk.rs`,
unchanged by the v4 cutover — still named with a `V3` suffix, since the row
shape itself didn't change, only where a node's embedding lives): `scope` is
the interned `ScopeRegistry` id, edge `from`/`to` are node **slots** (not
ULIDs), and label/edge-type/prop-key strings are `Dicts`-interned `u32`s.
Each row still carries its own ULID `id` — even though the table is
slot-keyed — so no extra table hop is needed to answer "what ULID is this
row." `disk::node_to_disk_v3`/`node_from_disk_v3` and
`edge_to_disk_v3`/`edge_from_disk_v3` convert to/from the public,
string-carrying `state::NodeRecord`/`EdgeRecord`, which are unchanged by the
v4 cutover. `disk::NodeRecordDisk`/`EdgeRecordDisk` (no `V3` suffix) are a
SEPARATE, frozen v2 ULID-keyed encode shape kept only because `migrate.rs`'s
v1→v2 step calls them; nothing in the live path reads or writes them.

Node disk rows carry no embedding, same as v3: `storage.rs: read_node`/
`read_node_by_slot` join the node's current embedding back onto the record
by slot before returning it — in v4 via `vector_store::read_vector_by_slot`
against `VECTORS`/`EMBEDDING_REF` (previously the cold `EMBEDDINGS` row) — so
an ordinary property edit never rewrites a node's (potentially large) vector.

## The `ops` table (unchanged since v1)

`OPS` is a u64-sequence-keyed table of raw postcard `Op` values (`op.rs`) —
never interned through `DICT`, never wrapped in the `codec.rs` value frame.
This is deliberate: the op log is self-describing, so replay
(`Storage::rebuild_state_from_ops`) never depends on what dictionary ids,
scope ids, or slots happened to be assigned when the log was first written.
Every other table in this document — dictionaries, the scope registry, slots,
adjacency, the prop index, the label index, the vector tables, and the FTS
tables — is *derived* state, reconstructible from `OPS` plus (for v1–v5
files) a one-time migration.

## Corruption contract

`storage.rs: read_node`/`read_edge` (and their `_by_slot` siblings' callers)
distinguish two failure shapes: no `NODE_SLOTS`/`EDGE_SLOTS` mapping at all is
an ordinary `Ok(None)` ("this id never existed"); a mapping that resolves to a
slot whose `NODES`/`EDGES` row is *absent* is `TopoError::Encoding` — never
silently treated as "not found." The mapping and its record row are written
and removed atomically on every code path, so they can only diverge if the
file itself is damaged, and that must surface loudly. `vector_store::
read_vector_by_slot` applies the identical two-cause split to `EMBEDDING_REF`
vs. `VECTORS` (see "Vectors," above).

## Rebuild determinism

`Storage::rebuild_state_from_ops` drains `NODES`, `EDGES`, `VECTOR_DIMS`,
`VECTORS`, `EMBEDDING_REF`, `DICT`, `SCOPES`, `NODE_SLOTS`, `NODE_IDS`,
`EDGE_SLOTS`, `EDGE_IDS`, `OUT_ADJ`, `IN_ADJ`, `PROP_INDEX`, `LABEL_INDEX`,
`POSTINGS`, `FTS_DOCS`, and `FTS_STATS` — every table this document calls
*derived* — then replays `OPS` from seq 1 through the same `apply_op` the
normal write path (`apply_batch`) uses, so there is no parallel mutation
logic to drift out of sync: `LABEL_INDEX` in particular is repopulated
exactly the way `apply_op`'s `CreateNode`/`RemoveNode` arms maintain it
incrementally on the live write path, not by a separate rebuild-only code
path. `COUNTERS` is preserved across the rebuild — access statistics
are auxiliary telemetry outside the op log, never appended to `OPS`, so a
rebuild must not reset them to zero — but preservation is keyed by **node
identity (ULID)**, not by slot number: replay reassigns dense slots in
*op-log* order, which need not match the slot order a table was in before
the rebuild (a migrated v2 file assigns slots in v2-ULID iteration order, for
one). `COUNTERS` is snapshotted by resolving each row's slot to its node's
ULID through the OLD `NODE_IDS` mapping before anything is drained, then
every surviving counter is re-inserted under that same node's NEW slot once
replay completes; a counter whose ULID no longer exists after replay is
dropped. `VECTOR_DIMS` is drained along with the other derived tables and
re-pinned entirely by replaying each `SetEmbedding` op through `apply_op`
again — replay does not special-case an already-seen `(model, dim)` pair, it
simply hits `check_or_pin_dim`'s `Ok` branch the same way a fresh write would.
Refuses with `TopoError::Compacted { oldest }` once `oldest_seq > 1`, since a
compacted log is no longer a full history and replay from genesis is
impossible by definition.

## Write atomicity (group commit)

`Storage::apply_batches` (F9c) applies every batch handed to it in one
"group" through a SINGLE shared write transaction and a SINGLE `tx.commit()`
(one fsync), rather than one transaction per batch — `apply_batch` is a thin
`apply_batches(vec![ops], now_ms)` wrapper, so a caller submitting one batch
at a time sees no change in behavior or durability. The applier's
drained-job path (`db.rs`) is what actually forms multi-batch groups: when
several `Job::Apply`s are already queued by the time the applier thread picks
up work, it drains and applies all of them as one group instead of one
transaction each.

This changes the crash-atomicity unit from "one batch" to "one group": if the
process is killed before that single `tx.commit()` returns, **every batch in
the group is lost together** — not just the one that happened to be
mid-flight — and on reopen the log holds exactly the pre-group state, same as
any other transaction that never committed (see "Version and migration,"
above, for the identical single-commit argument applied to migration). A
crash can never leave the group PARTIALLY applied: `apply_ops_in_txn` mutates
the shared transaction's tables incrementally as it walks the group, but
nothing in that transaction is durable — visible to a REOPENED file — until
the one `tx.commit()` at the end succeeds. This is coarser atomicity than
per-batch commit, deliberately: grouping trades "a crash loses at most one
batch" for "a crash loses at most one group, but N batches share one fsync,"
and the direction of that trade is a hard invariant — a future change may
grow the group (coarser still), but must never commit part of a group without
committing all of it (finer than one atomic unit per group is the one thing
this design forbids). Concurrent readers are unaffected either way: they see
either the pre-group state or the fully-committed post-group state, never
an in-between (`apply_batches`' in-memory `Dicts`/`ScopeRegistry` mirror
guards are held across the whole group and dropped only once every batch has
resolved successfully, immediately before the commit — see `apply_batches`'
doc comment in `storage.rs`).

## Evolution policy

On-disk serde enum variants (`Op`, `PropValue`, `IndexValue`, ...) are
append-only. In addition, four plain byte registries are **APPEND-ONLY** —
an existing value's meaning may never be repurposed, only new values added:

- `dict.rs: DictKind` — `0x00` Label, `0x01` EdgeType, `0x02` PropKey, `0x03`
  Model (v4).
- `codec.rs` value frame — `0x00` raw postcard, `0x01` unused/reserved,
  `0x02` lz4.
- `adj.rs: ADJ_BLOCK_FORMAT_V0` — currently the only defined adjacency block
  format byte (`0x00`); `decode_block` rejects any other value.
- `fts.rs: POSTINGS_BLOCK_FORMAT_V0` (v4) — currently the only defined
  chunked-postings block format byte (`0x00`); `decode_posting_block`/
  `posting_block_count` reject any other value.

The op log deliberately retains full strings and no frame so replay stays
independent of dictionary/scope-registry assignment order (see "The `ops`
table," above). `Storage::rebuild_state_from_ops` drains and regenerates every
derived table — dictionaries, the scope registry, slots, adjacency, the prop
index, the label index, the vector tables, and FTS — while replaying this
self-describing log (see "Rebuild determinism," above).
